## 🌳 Дерево прав (Role Tree) — КЛЮЧЕВАЯ концепция

**Это НЕ оргструктура компании, это КЛАССИФИКАЦИЯ РОЛЕЙ!**

### Структура
- Дерево редактирует **только директор** (супер-пользователь, корень дерева)
- Каждому сотруднику назначается **один узел** дерева (= его должность/право)
- **Поддерево пользователя** = его узел + все узлы ниже
- При создании задачи список исполнителей фильтруется только людьми **из поддерева** создателя

### Пример дерева
```
👑 Директор (корень, супер-права)
├── 💼 Руководитель отдела продаж
│   ├── 📊 Менеджер по продажам
│   └── 🎯 Стажёр
├── 🧑‍💻 Руководитель IT
│   ├── 👨‍💻 Senior
│   └── 👩‍💻 Junior
└── 🎨 Арт-директор
    └── 🖌️ Дизайнер
```

### Правила
- 👑 Директор = корень, видит всё, может всё (как у дерева корень всегда существует, даже у вырожденного в один узел случая)
- 🔍 "Поиск в поддереве" — SQL: `WHERE role_node_id IN (...subtree_ids...)`
- 👤 Один человек = один узел (без двойного подчинения)
- 🔧 При увольнении — "передать поддерево" другому узлу (узел не удаляется, меняется привязка пользователя)
- 🔮 Будущее: визуальный drag-and-drop редактор дерева

### Технические детали
- Таблица `role_tree`: `id`, `parent_id`, `title`, `description`, `level`, `created_by`
- Таблица `user_role_assignments`: `user_id`, `role_node_id`, `assigned_at`, `assigned_by`
- Все операции с деревом только через транзакции
- Рекурсивный CTE для получения поддерева

## 📋 Задачи (Tasks)

### Структура задачи
| Поле | Тип | Описание |
|------|-----|----------|
| title | string | Название |
| description | text | Описание от создателя (редактируется только им) |
| importance | enum | `green` / `yellow` / `red` (светофор) |
| hard_deadline | timestamp | **Твёрдый дедлайн** — когда должно быть выполнено |
| checkpoints[] | array | **Контрольные точки** (дата + название) — промежуточные проверки |
| assignee_ids[] | array | Исполнители — из поддерева создателя |
| watcher_ids[] | array | Смотрящие (опционально). Создатель по умолчанию "смотрящий" |
| files[] | array | Прикреплённые файлы (результаты работы) |
| status | enum | new / in_progress / on_review / done / overdue / rejected / archived |
| executor_comment | text | **Общий блокнот команды** — пишут все исполнители, видят все |
| watcher_comment | text | Заметки смотрящего о проверке |
| archived_as | enum | done / removed (если создатель "удалил как нерелевантную") |

### Жизненный цикл
1. Создатель создаёт → статус `new`
2. Исполнитель(и) начинают работать → `in_progress`
3. На контрольной точке смотритель проверяет → `on_review`
4. Смотритель подтверждает/отклоняет:
   - ✅ `done` (выполнено)
   - ↩️ `rejected` (на доработку, с комментарием)
5. Создатель архивирует → `archived` (или смещает дедлайн)

### Внутри задачи — 3 разделённых поля
```
┌─ Заголовок + мета ──────────────────────┐
│ [Важность] [Дедлайн] [Создатель]        │
└──────────────────────────────────────────┘

┌─ 👤 Поле СОЗДАТЕЛЯ ─────────────────────┐
│ Описание задачи (только он редактирует) │
│ "Продать 1000 тонн арбузов.             │
│  Результат: отчёт + фото отгрузок"      │
└──────────────────────────────────────────┘

┌─ 👥 Поле ИСПОЛНИТЕЛЕЙ (Shared Canvas) ──┐
│ Общий блокнот команды                   │
│ • Рома: "Предлагаю 3 канала"            │
│ • Вася: "У меня база 500 лидов"         │
│ [+ Добавить мысль]                      │
└──────────────────────────────────────────┘

┌─ 👁️ Поле СМОТРЯЩЕГО ────────────────────┐
│ Заметки о проверке                      │
└──────────────────────────────────────────┘
```

**Что это значит:**
- **Поле создателя** = описание задачи (редактирует только создатель)
- **Поле исполнителей (Shared Canvas)** = общий блокнот команды, пишут ВСЕ исполнители, видят ВСЕ участники. Это и есть "доска"!
- **Поле смотрящего** = его заметки о проверке
- **Статус** = меняют: исполнитель (→ "в работе"/"на проверку"), смотрящий (→ "выполнено"/"отклонено"), создатель (→ "в архив"/"сместить дедлайн")

### Сортировка списка задач (3 вида)
1. **По дедлайну** (приоритет) → важность
2. **По важности** (приоритет) → дедлайн
3. **Календарь** (позже): задачи как "змейки" на сетке, с точками-КТ

### Технические детали
- Таблица `tasks`: `id`, `title`, `description`, `importance`, `hard_deadline`, `status`, `creator_id`, `executor_comment`, `watcher_comment`, `archived_as`
- Таблица `task_assignees`: `task_id`, `user_id`
- Таблица `task_watchers`: `task_id`, `user_id`
- Таблица `task_checkpoints`: `id`, `task_id`, `title`, `deadline`, `status`
- Таблица `task_canvas_posts`: `id`, `task_id`, `author_id`, `content`, `created_at` (Shared Canvas)
- Таблица `task_files`: `id`, `task_id`, `file_url`, `file_name`, `uploaded_by`, `uploaded_at`
- WebSocket для real-time обновлений Shared Canvas

## 🗒️ Заметки (Notes)

- Личные заметки (как Apple Notes)
- Содержимое: текст + чеклист + изображения + файлы
- Организация: **цветные теги** (как Google Keep) + папки (как Apple Notes) + поиск
- **Шеринг** в мессенджер (как "поделиться" в Telegram):
  - Выбираешь чат → приходит карточка-превью "Ваня поделился заметкой 'План на квартал'" + кнопка "Принять"
  - Принявший получает **независимую копию** (изменения у отправителя не синхронизируются)

### Технические детали
- Таблица `notes`: `id`, `user_id`, `title`, `content`, `color_tag`, `folder_id`, `created_at`, `updated_at`
- Таблица `note_folders`: `id`, `user_id`, `name`, `color`
- Таблица `note_attachments`: `id`, `note_id`, `file_url`, `file_name`, `uploaded_at`
- Таблица `note_shares`: `id`, `note_id`, `from_user_id`, `to_user_id`, `accepted`, `accepted_at`
